# MoveIt 2 Collision-Aware Motion Planning for a 4-DOF Robotic Arm

## Prerequisites

- Ubuntu 22.04
- ROS 2 Humble
- MoveIt 2

## Problem Statement

The objective of this task was to demonstrate collision-aware motion planning using MoveIt 2 on a custom 4-DOF robotic arm in ROS 2 Humble. The specific requirements were as follows:

1. Define a custom robotic arm using a URDF model with proper visual and collision geometry.
2. Configure MoveIt 2 for the arm using the MoveIt Setup Assistant, including planning groups, self-collision matrices, virtual joints, and end-effectors.
3. Spawn a collision object (a cube) into the planning scene at runtime.
4. Command the arm to move to a specific Cartesian (x, y, z) target such that MoveIt 2 is forced to plan a trajectory that avoids the collision object.
5. Write a custom C++ ROS 2 node that programmatically performs steps 3 using the MoveIt 2 C++ API.

The deliverables were two ROS 2 packages:
- `mybot_description` -- the robot model and the custom C++ manipulation node.
- `mybot_moveit_config` -- the MoveIt 2 configuration generated by the Setup Assistant.

---

## Approach

The task was broken down into the following stages:

1. **Robot Modelling**: Start with the existing `mybot.urdf` file. Ensure every link has both `<visual>` and `<collision>` tags, and inject `<ros2_control>` hardware interface definitions so that the MoveIt demo launch can instantiate a simulated controller.

2. **MoveIt 2 Configuration**: Use the MoveIt Setup Assistant to load the URDF, define planning groups (`arm` and `gripper`), set up a virtual joint to anchor the robot to the world frame, configure the end-effector, and generate the self-collision matrix. Export the configuration as the `mybot_moveit_config` package.

3. **Configuration Refinement**: Manually fix several configuration files that the Setup Assistant generated incorrectly or incompletely. This included correcting data types in `joint_limits.yaml`, populating empty interface arrays in `ros2_controllers.yaml`, adding missing fields in `moveit_controllers.yaml`, and enabling `position_only_ik` in `kinematics.yaml` for the under-actuated arm.

4. **Custom Node Development**: Write a C++ node (`urc_manipulation.cpp`) inside `mybot_description` that connects to the running `move_group` server, fetches the robot description parameters, and spawns a collision cube into the planning scene. Motion planning around the obstacle is then performed interactively via RViz.

5. **Integration and Testing**: Build both packages with `colcon`, launch the MoveIt demo in one terminal (which starts `move_group`, RViz, and the controller manager), and run the custom node in a second terminal to verify obstacle avoidance.

## Task Workflow

This section describes the end-to-end execution flow when the system is running.

### Step 1: Launch the MoveIt Demo

```bash
cd ~/mybot_ws
source install/setup.bash
ros2 launch mybot_moveit_config demo.launch.py
```

This single command triggers the following chain of events:

1. The `MoveItConfigsBuilder` in `demo.launch.py` reads the package name `mybot_moveit_config` and the robot name `arm`, then automatically locates and loads the URDF (`mybot.urdf`), the SRDF (`arm.srdf`), the kinematics solver configuration (`kinematics.yaml`), the joint limits (`joint_limits.yaml`), and the controller definitions (`ros2_controllers.yaml`, `moveit_controllers.yaml`).
2. The `move_group` node starts. It parses the URDF to build an internal kinematic model of the robot, parses the SRDF to learn about planning groups and disabled collision pairs, initialises the OMPL motion planning pipeline, and advertises ROS 2 action servers for planning and execution.
3. The `controller_manager` node starts and loads the `arm_controller` and `gripper_controller` as `JointTrajectoryController` instances using the `mock_components/GenericSystem` hardware plugin defined in the URDF. This simulates a real robot without requiring physical hardware.
4. The `joint_state_broadcaster` begins publishing `/joint_states` so that RViz and the planning scene stay synchronised with the simulated joint positions.
5. RViz opens with the MoveIt MotionPlanning plugin pre-loaded. It subscribes to the `robot_description` parameter and `/joint_states` to render the robot model and the interactive planning markers.

### Step 2: Run the Custom Manipulation Node

```bash
cd ~/mybot_ws
source install/setup.bash
ros2 run mybot_description urc_manipulation
```

This triggers the following sequence inside `urc_manipulation.cpp`:

1. The node initialises and spins a background executor in a detached thread so that ROS 2 callbacks (service responses, action feedback) are processed concurrently.
2. A separate temporary node (`urc_manipulation_param_fetcher`) creates a `SyncParametersClient` targeting the `move_group` node. It waits for the parameter service to become available, then requests `robot_description`, `robot_description_semantic`, and the kinematics solver parameters.
3. Each fetched parameter is declared on the primary node with the correct type (string or double). This step is mandatory in ROS 2 Humble because parameters must be declared before they can be accessed.
4. A `MoveGroupInterface` is constructed for the `arm` planning group. Internally this creates an action client that connects to the `move_group` action server.
5. A `PlanningSceneInterface` is constructed. This provides a convenient API for adding, removing, or attaching collision objects.
6. A `CollisionObject` message is built: a 0.2 m x 0.2 m x 0.2 m cube placed at (0.40, 0.0, 0.25) in the planning frame. The object is pushed into the planning scene via `applyCollisionObjects`.
7. The node sleeps for one second to allow the scene update to propagate to the `move_group` server.
8. 8. At this point the obstacle is visible in RViz as a green solid. The user can then use the RViz MotionPlanning plugin to interactively set a pose target and plan around the obstacle.

### Step 3: Observe in RViz

Once both terminals are running, RViz displays:
- The robot arm in its current configuration (grey model).
- The collision cube (green box) in front of the robot.
- Interactive markers on the arm tip that allow manual goal-state selection.
- - The planned trajectory (orange trail) that curves around the obstacle when Plan and Execute is clicked in the MotionPlanning plugin.

## Implementation Details

This section describes the specific problems encountered during the task and the solutions applied.



### 1. Integer vs Double Type Mismatch in joint_limits.yaml

The MoveIt Setup Assistant generated `joint_limits.yaml` with integer values (e.g., `max_velocity: 1` instead of `max_velocity: 1.0`). ROS 2 Humble strictly enforces parameter types, and the `move_group` node crashed with `InvalidParameterTypeException: expected [double] got [integer]`.

**Solution**: All numeric values in `joint_limits.yaml` were changed to explicit floating-point notation (e.g., `1.0`, `0.0`).

### 2. Empty Controller Interfaces in ros2_controllers.yaml

The generated `ros2_controllers.yaml` had empty arrays for `command_interfaces` and `state_interfaces`. The `controller_manager` requires at least one command interface and one state interface to load a controller.

**Solution**: The arrays were populated:
```yaml
command_interfaces:
  - position
state_interfaces:
  - position
  - velocity
```

### 3. KDL IK Solver Timeout on 4-DOF Arm

The arm has only four joints (hip, shoulder, elbow, wrist). When a Cartesian (x, y, z) target was set via `setPositionTarget`, the KDL inverse kinematics solver attempted to satisfy both position and orientation constraints. Since a 4-DOF arm cannot achieve arbitrary 6-DOF orientations, the solver looped until it timed out.

**Solution**: The flag `position_only_ik: true` was added to `kinematics.yaml`. This instructs KDL to solve only for position (x, y, z), ignoring orientation constraints entirely. The solver timeout was also increased from 0.005 s to 0.05 s to give the solver sufficient iterations.

### 4. Executor Conflict with SyncParametersClient

When the custom node used `SyncParametersClient` on the same node that was already spinning in a background executor, it caused the runtime error `Node '/urc_manipulation' has already been added to an executor`. This occurred because `SyncParametersClient` internally creates its own executor and attempts to add the same node to it.

**Solution**: A second, dedicated `rclcpp::Node` named `urc_manipulation_param_fetcher` was created solely for the parameter client. This node is never added to the background executor, so there is no conflict.


### 5. Wayland Display Server Crash

ROS 2 GUI applications (RViz, MoveIt Setup Assistant) default to the Wayland Qt backend on modern Ubuntu installations. This caused segmentation faults on launch.

**Solution**: The environment variable `QT_QPA_PLATFORM=xcb` was added to `~/.bashrc` and `~/.zshrc` to force the X11 backend.

## Key Learnings

### MoveIt 2 Architecture

MoveIt 2 is a motion planning framework that sits on top of ROS 2. It consists of three core components:
- **move_group**: The central planning server. It loads the robot model (URDF + SRDF), initialises one or more planning pipelines (typically OMPL), maintains the planning scene (current joint states plus known obstacles), and exposes action servers for planning and execution.
- **PlanningSceneInterface**: A client-side API for modifying the environment. Collision objects can be added, removed, or attached to robot links at runtime.
- **MoveGroupInterface**: A client-side API for requesting motion plans. It wraps the action client that communicates with `move_group`. Users can set joint targets, pose targets, or Cartesian path constraints and then call `plan()` and `execute()`.

### MoveIt Setup Assistant

The Setup Assistant is a GUI tool that generates the `mybot_moveit_config` package. The key screens and their outputs are:

| Screen | What It Generates |
|--------|-------------------|
| Start | Loads the URDF and validates the robot model. |
| Self-Collisions | Generates the `<disable_collisions>` entries in the SRDF by running a random sampling algorithm that checks every pair of links for default/adjacent/never collisions. |
| Virtual Joints | Creates a `<virtual_joint>` entry that anchors the robot's base to a world frame. Required for fixed-base manipulators. |
| Planning Groups | Defines `<group>` entries in the SRDF. A group can be specified as a kinematic chain (base link to tip link), a list of joints, or a list of links. |
| Robot Poses | Defines `<group_state>` entries (named joint configurations such as `home`). |
| End Effectors | Defines `<end_effector>` entries that associate a gripper group with a parent link on the arm group. |
| ROS 2 Controllers | Generates `ros2_controllers.yaml` and `moveit_controllers.yaml`. |
| Launch Files | Generates `demo.launch.py` and supporting launch files. |
| Configuration Files | Writes all generated files to the output package directory. |

### Path Planning and OMPL

MoveIt 2 uses the Open Motion Planning Library (OMPL) as its default planning pipeline. The planning process works as follows:

1. The user sets a goal (joint-space target or Cartesian pose target).
2. If the goal is a Cartesian pose, the inverse kinematics solver (KDL in this project) converts it to a joint-space goal. For under-actuated arms (fewer than 6 DOF), `position_only_ik: true` must be enabled so the solver ignores orientation.
3. OMPL samples random configurations in the joint space and builds a search tree (using algorithms such as RRT, RRT-Star, or PRM).
4. At each sample, OMPL queries the planning scene for collision checks. If a sampled configuration causes the robot to intersect with any collision object (including the self-collision matrix), the sample is discarded.
5. Once a valid path from the start state to the goal state is found, OMPL returns the path as a sequence of waypoints.
6. MoveIt applies time parameterisation to the waypoint sequence, converting it into a smooth trajectory that respects velocity and acceleration limits defined in `joint_limits.yaml`.

### Inverse Kinematics (IK)

Inverse kinematics is the process of computing joint angles that place the end-effector at a desired position and orientation. In this project the KDL (Kinematics and Dynamics Library) plugin was used. Key observations:

- KDL is a numerical iterative solver. It starts from the current joint configuration and iteratively adjusts joint angles to minimise the error between the current end-effector pose and the target pose.
- For a 4-DOF arm, a full 6-DOF pose target (position + orientation) is under-constrained. The solver cannot find a solution because there are not enough joints to satisfy all six constraints simultaneously. Setting `position_only_ik: true` reduces the problem to 3 constraints (x, y, z), which the 4-DOF arm can satisfy.
- The `kinematics_solver_timeout` parameter controls how long the solver iterates before giving up. A value too small (e.g., 0.005 s) causes premature failures; 0.05 s was sufficient for this arm.
- The `kinematics_solver_search_resolution` parameter controls the discretisation of the joint space when searching for IK solutions. A smaller value increases accuracy but takes longer.

### ROS 2 Control

ROS 2 Control is a hardware abstraction framework that separates the controller logic from the hardware driver. In this project:

- The URDF contains a `<ros2_control>` block that declares a `mock_components/GenericSystem` hardware plugin. This plugin simulates a real robot by accepting position commands and reporting back simulated joint states.
- The `controller_manager` node loads and manages controllers at runtime. It reads `ros2_controllers.yaml` to determine which controllers to instantiate and which joints they control.
- `JointTrajectoryController` is the controller type used for both the arm and the gripper. It receives a `FollowJointTrajectory` action goal (a time-parameterised sequence of joint positions) and forwards position commands to the hardware interface at the configured update rate (100 Hz).
- The `joint_state_broadcaster` reads joint states from the hardware interface and publishes them on `/joint_states` so that RViz and the planning scene remain synchronised.

### MoveIt Controllers vs ROS 2 Controllers

These two configuration files serve different layers of the system:

- `ros2_controllers.yaml` configures the **ros2_control layer**. It defines the controller types, the joints they manage, and the hardware command/state interfaces. This is what the `controller_manager` reads.
- `moveit_controllers.yaml` configures the **MoveIt layer**. It tells MoveIt the names of the controllers, their action namespace (`follow_joint_trajectory`), and which joints they control. MoveIt uses this information to route planned trajectories to the correct controller via the `FollowJointTrajectory` action interface.

The two files must be kept in sync: the controller names, joint lists, and interface types must match. If they diverge, MoveIt will either fail to find a controller for the planned trajectory or send it to the wrong one.
